本篇文章修改參考[這篇](https://www.cyut.edu.tw/~ckhung/b/c/gdb.php)文章製作範例和教學，首先我們先寫一個有bug的selection sort程式`ssort_bug.c`

```c
#include <stdio.h>
#include <stdlib.h>

int select_min_index(double data[], int N);
void swap(double x, double y);

int main(int argc, char *argv[]) {
    int i, min;
    double data[8];

    for (i=1; i<=argc; ++i) {
	data[i] = atof(argv[i]);
    }
    for (i=1; i<=argc; ++i) {
	min = select_min_index(&data[i], argc-i);
	swap(data[i], data[min]);
    }
    for (i=1; i<=argc; ++i) {
	printf(" %f", data[i]);
    }
    printf("\n");
    return 0;
}

int select_min_index(double data[], int N) {
    int j, min = 0;
    for (j=1; j<N; ++j) {
	if (data[j] < data[min])
	    min = j;
    }
    return min;
}

void swap(double x, double y) {
    double t;
    t = x;
    x = y;
    y = t;
}
```
{: file='ssort_bug.c'}

接下來用gcc編譯程式並且執行
```bash
gcc -Wall -o ssort ssort_bug.c  # 編譯
./ssort 12.3 -17 6.5  # 執行
```
執行後應該會出現`Segmentation fault`錯誤，也看不出問題出在哪裡。

# gdb除錯
## run
首先使用gdb執行我們的程式，並且在gdb的terminal直接輸入我們要給ssort程式的參數。
```bash
gdb ssort
run 12.3 -17 6.5 # terminal開頭出現(gdb)的時候代表進入gdb的terminal了
```

我們可以看到類似下面的訊息
```bash

[Thread debugging using libthread_db enabled]
```

## where指令
首先確認程式執行到何處才出錯，輸入`where`指令可以看到call stack類似下面訊息，數字越小的被執行的時間越靠近現在，因此可以得知程式執行到`__GI_____strtod_l_internal`出現錯誤了
```
#0  0x00007ffff7dc7773 in __GI_____strtod_l_internal (nptr=0x0, endptr=0x0, group=<optimized out>, loc=0x7ffff7f96580 <_nl_global_locale>)
    at ./stdlib/strtod_l.c:610
#1  0x00005555555551f3 in main ()
```

## list、l指令
接下來我們想看看錯誤出現在我們程式碼的哪一段，於是我們接著下`list`指令或是`l`指令，不過我們可以發現我們的程式碼沒有被顯示出來，因為剛剛編譯的時候沒有加上`-g`這個選項，所以`list`指令無效。`-g`這個選項告訴編譯器把debug所需的資訊都帶進我們的執行檔裡面。  
<br />
注意!!原本的gdb視窗不需要關掉，我們只需要另外開一個terminal視窗重新編譯就好，這對於除錯大型程式非常重要，因為我們可能已經埋了許多中斷點，如果關掉gdb重來的話非常浪費時間。  
我們開一個新的terminal視窗並輸入新的編譯指令
```bash
gcc -Wall -g -o ssort ssort_bug.c
```
接著回到原本的gdb視窗下`run`指令，而且這次可以不必給參數，gdb會自動帶入上次給的參數，gdb會問你要不要重頭在執行一次，選擇`y`，接著你可以注意到這次gdb會告訴你執行檔被更動過了。
```bash
ssort has changed; re-reading symbols.
```
接著再次輸入`where`指令，你可以比較一下main函式那行跟之前沒有使用`-g`編譯我們多了一些訊息
1. 多了輸入main函式的參數
2. 多了每一個函式被執行到哪一行，例如我們的範例main()函式執行到`ssort_bug.c`的第12行後，進入下一個函式`__GI_____strtod_l_internal`而我們的程式就在這個函式裡故障。當然我們應該先合理懷疑錯誤是出現在我們的`ssort_bug.c`第12行，因為`__GI_____strtod_l_internal`是外部函式庫，相較於我們的程式出錯的機率低比較多。
```bash
#0  0x00007ffff7dc7773 in __GI_____strtod_l_internal (nptr=0x0, endptr=0x0, group=<optimized out>, loc=0x7ffff7f96580 <_nl_global_locale>)
    at ./stdlib/strtod_l.c:610
#1  0x00005555555551f3 in main (argc=4, argv=0x7fffffffddd8) at ssort_bug.c:12
```
